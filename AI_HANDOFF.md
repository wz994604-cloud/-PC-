# 28LIVE AI 接手说明

## 给新的 AI

你接手的是一个已经暂停开发的真实项目，不是从零搭建任务。先阅读本文件、`PROJECT_OVERVIEW.md`、`PROJECT_STATUS.md`、`README.md`、`research/README.md`，再查看代码。不要重新设计页面，不要重新实现 v0.1，不要直接部署，也不要根据旧文档中某一次部署记录推断当前外部状态。

## 项目背景

用户要重建一个可维护、可测试、可部署的 28LIVE 加拿大 28 开奖平台。项目经历了开奖核心、移动端页面、预测历史、v0.1 模型和离线回测研究几个阶段。用户最重视三件事：

1. 开奖数据必须真实，和值、大小和单双由本项目自行校验。
2. 页面在手机端稳定，既有布局和视觉不要反复改变。
3. 预测必须在开奖前固化、同一期不可重复或覆盖，不能事后补造历史来美化命中率。

项目现已进入归档交接阶段。用户要求完整源码上传 GitHub，完成交接后暂停。

## 用户真正想实现什么

最终目标不是一个静态展示页，而是一套可信的开奖与预测记录系统：

- 稳定读取并展示最新与历史开奖。
- 可靠地在每期开奖前生成下一期预测并永久保存。
- 开奖后自动核对当期预测，但不修改原预测内容。
- 能长期积累真实历史，用锁定测试集客观比较模型。
- 只有经过 Preview 连续观察和完整测试的版本才允许进入 Production。

当前代码完成了大部分应用逻辑，但“永久保存”和“可靠的开奖前调度”仍未达到 Production 要求。

## v0.1 模型结构和权重

v0.1 对和值 0–27 生成完整分布，使用三个分量：

| 分量 | 权重 | 当前实现 |
|---|---:|---|
| 理论分布 | 25% | 枚举三个 0–9 数字的 1000 种组合得到和值理论分布 |
| 近期频率 | 55% | 对已保存历史和值计数后归一化 |
| 遗漏 | 20% | 按各和值距最近出现的位置构造遗漏分量并归一化 |

系统读取最多 300 期历史，按期号倒序计算，推荐综合概率最高的和值。它还输出综合评分、信心指数、风险标签、样本量和各和值分布。

重要行为：

- 无历史时返回 `null`，页面/API 表示数据积累中。
- 模型无随机数，相同历史输入产生相同输出。
- 下一期号通过最新期号加 1 得到。
- `prediction_history.issue` 是主键。
- 首次插入使用 `ON CONFLICT(issue) DO NOTHING`。
- 开奖后只填写 `actual_sum`、`hit`、`checked_at`，不覆盖推荐和值、分布、权重或生成时间。

不要修改上述结构和权重，除非用户在恢复开发后明确开启新模型阶段。

## v0.2 的设计方向

v0.2 目前名为 `v0.2-candidate-a`，只存在于 `research/`。设计方向是避免简单追热或追遗漏，通过理论分布收缩降低小样本噪声，并组合：

- 理论分布。
- 20、50、100 期频率。
- 归一化遗漏，遗漏比上限为 3。
- 冷热变化。
- 一阶转移分布。
- 趋势、动量和波动。
- 大小、单双及反转状态信号。
- 样本不足时关闭或降低不可靠特征权重。

当前配置位于 `research/config.ts`。默认基础权重为理论 30%、20 期频率 12%、50 期频率 18%、100 期频率 12%、遗漏 12%、冷热变化 7%、转移 4%、趋势/动量/波动 3%、状态信号 2%。由于某些特征在样本不足时关闭，未使用权重会回到理论分布。

这些权重只是研究候选，不是线上参数，不得复制到 `src/lib/prediction/model.ts`。

## 回测框架已经完成哪些内容

`research/` 与 `src/app` 没有导入关系，线上 API 不会调用回测代码。框架已经支持：

- 从 SQLite `draw_records`、真实 CSV 或 JSON 读取数据。
- 检查重复期号、时间逆序、非法号码、非法和值和样本量。
- theoretical-baseline、theoretical-mode-baseline、v0.1、v0.2-candidate-a 统一模型接口。
- 扩展窗口和固定滚动窗口。
- 100 期默认热身。
- 按时间 60% 训练、20% 验证、20% 锁定测试。
- 确保第 t 期预测只读取 t-1 及以前数据。
- 输出 Top1/3/5、推荐和值、实际和值、MAE、Brier、Log Loss、ECE、熵和分组统计。
- 在真实样本少于 100 期时只做数据质量检查，不输出模型优劣结论。
- 自动化测试验证防未来数据泄漏和配置行为。

当前仓库没有足够真实历史样本，不能产生有效的 v0.1 与 v0.2 比较结论。不要生成模拟数据冒充真实历史，也不要用锁定测试集调参。

## 为什么目前不能直接部署 Production

1. 托管环境使用 `/tmp/28live.sqlite`，预测历史可能在实例重启、切换实例或重新部署后清空。
2. SQLite 在 Serverless 多实例之间不共享，无法保证长期唯一和连续历史。
3. Vercel Hobby 每分钟 Cron 已移除；当前访问触发只能作为补偿，无法保证无人访问时每期在开奖前执行。
4. Preview 的三个核心接口、线上重复请求幂等和页面一致性尚未完成最终验收。
5. 连续 3 个真实开奖周期观察尚未完成。
6. v0.2 没有足够真实数据、没有锁定测试集优势证据，不能上线。

因此当前代码“可运行、可测试、可构建”不等于“具备 Production 发布条件”。

## 所有已知问题

- 接口访问触发会在无人访问时延迟累计和核对。
- 正式调度必须保证预测在开奖前生成；开奖后不能补造当期预测。
- Preview 受 Vercel Authentication 保护，本轮在线验收未完成。
- 连续 3 期观察未完成。
- 当前没有足够真实历史数据比较模型。
- 旧 `PROJECT_STATE.md` 和 `PROJECT_HANDOFF.md` 记录过不同部署与观察结果，存在时序矛盾。当前状态以 `PROJECT_STATUS.md` 为准，外部部署状态恢复时必须重新只读确认。
- 内部累计接口要求 `CRON_SECRET`，但仓库不得包含其真实值。
- E2E 依赖浏览器环境；如果 Chromium/WebKit 不可用，不能把未执行误报为通过。

## 恢复开发时的正确顺序

### 第一步：解决预测历史持久化，接入远程数据库

- 选择适合 Vercel Serverless 的远程数据库，例如托管 PostgreSQL。
- 迁移 `draw_records` 与 `prediction_history` 的语义和历史。
- 以“期号 + 模型版本”或经确认的等价唯一约束保证幂等。
- 使用事务保证首次固化、开奖核对和并发请求安全。
- 已固化预测字段不可更新。
- 保留本地 SQLite 作为开发/测试适配器时，线上必须明确使用远程数据库。
- 任何连接字符串只放环境变量，禁止提交仓库。

### 第二步：完成 Preview 在线接口验收和连续 3 期观察

- 部署到用户指定且有权限的 Preview 项目。
- 验证 `/api/source`、`/api/prediction`、`/api/prediction/history`。
- 验证页面布局和功能无变化。
- 重复调用验证同一期不重复、不覆盖。
- 验证重新部署或实例变化后历史仍存在。
- 连续观察至少 3 个真实开奖周期，确认每期预测在开奖前固化、开奖后只核对。
- 若使用外部调度，接口必须密钥保护，并保留访问触发作为补偿；不得恢复 Hobby 不允许的每分钟 Vercel Cron。

### 第三步：确认 v0.1 稳定后，再决定是否部署 Production

- 完成全部本地测试、lint、typecheck、build 和必要 E2E。
- 完成 Production 发布前检查清单。
- Production 只能使用 v0.1。
- 任一核心接口异常立即停止，不继续发布。

### 第四步：积累足够真实历史数据后，再比较 v0.1 和 v0.2

- 使用永久保存的真实、按时间排序数据。
- 保留训练/验证/锁定测试边界。
- 不补造缺失预测，不使用未来数据，不用锁定测试集调权重。
- 报告样本量、数据质量和统计不确定性。

### 第五步：只有 v0.2 在锁定测试集上稳定优于 v0.1，才允许考虑上线

- 优势必须跨多个指标和合理时间窗口稳定存在。
- 先作为独立 Preview 候选运行，不直接替换 v0.1。
- 完成影子观察、回滚方案和用户明确批准后，才可讨论 Production。

## 接手后的第一条原则

先解决数据真实性与持久化，再验收运行稳定性，最后才研究模型升级。不要把模型复杂度放在记录可信度之前。
